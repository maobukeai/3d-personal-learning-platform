# 3D 学习平台部署成果与热部署更新指南 🚀

这份文档用于两件事：

1. 第一次把 3D 学习平台部署到服务器。
2. 后续用一条命令完成热部署更新。

当前推荐部署目录：

```bash
/var/www/3d-lms
```

当前推荐域名示例：

```bash
https://lilan8.cn
```

如果你的服务器、域名或数据库名称不同，把命令里的对应值替换掉即可。

***

## 置顶：最常用复制命令

### 1. 日常一键热部署

本地代码已经推送到 GitHub 后，服务器只需要执行：

```bash
cd /var/www/3d-lms
chmod +x fast_update.sh deploy.sh
./fast_update.sh
```

### 2. 旧版脚本故障恢复

如果服务器执行 `./fast_update.sh` 时出现大量：

```text
file has vanished
rsync warning: some files vanished before they could be transferred (code 24)
```

复制下面整段命令恢复一次：

```bash
cd /var/www/3d-lms

rm -rf /tmp/3d-lms-update update.zip .update_extract
mkdir -p /tmp/3d-lms-update

wget -T 20 -O /tmp/3d-lms-update/update.zip \
  "https://gh-proxy.com/https://github.com/maobukeai/3d-personal-learning-platform/archive/refs/heads/main.zip"

unzip -q /tmp/3d-lms-update/update.zip -d /tmp/3d-lms-update

rsync -a --delete \
  --exclude '.git/' \
  --exclude '.env' \
  --exclude 'server/.env' \
  --exclude 'node_modules/' \
  --exclude 'server/node_modules/' \
  --exclude 'logs/' \
  --exclude 'server/uploads/' \
  /tmp/3d-lms-update/3d-personal-learning-platform-main/ \
  /var/www/3d-lms/

chmod +x /var/www/3d-lms/deploy.sh /var/www/3d-lms/fast_update.sh

if ! grep -q 'SKIP_TYPECHECK' /var/www/3d-lms/deploy.sh; then
  sed -i '0,/npm run build/s//npx vite build/' /var/www/3d-lms/deploy.sh
fi

SKIP_TYPECHECK=1 NODE_BUILD_MEMORY_MB=1536 /var/www/3d-lms/deploy.sh
```

## 一、当前线上架构

| 模块    | 推荐配置                              | 说明                         |
| :---- | :-------------------------------- | :------------------------- |
| 前端    | Nginx 托管 `/var/www/3d-lms/dist`   | Vue 3 + Vite 打包产物          |
| 后端    | PM2 运行 `server/dist/src/index.js` | 默认端口 `3001`                |
| 数据库   | MySQL                             | 推荐数据库名 `learning_platform` |
| 进程管理  | PM2                               | 支持崩溃自动拉起、平滑重载              |
| 热部署   | `./fast_update.sh`                | 多备用加速节点下载 GitHub main.zip  |
| 小内存保护 | swap + 跳过服务器 `vue-tsc`            | 适合 2G 左右内存的轻量云服务器          |

## 二、第一次部署要做什么

第一次部署和后续热部署不一样。第一次服务器上还没有代码、依赖、数据库、环境变量、管理员账号，所以要先完成初始化。

### 1. 登录服务器

```bash
ssh root@你的服务器公网 IP
```

进入部署目录：

```bash
mkdir -p /var/www/3d-lms
cd /var/www/3d-lms
```

### 2. 安装基础软件

Ubuntu / Debian 系统可以使用：

```bash
apt update
apt install -y curl wget unzip rsync nginx mysql-server
```

检查 Node：

```bash
node -v
npm -v
```

项目推荐 Node 20。如果服务器没有 Node 20，可以安装：

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt install -y nodejs
node -v
```

安装 PM2：

```bash
npm install -g pm2
```

### 3. 准备 MySQL 数据库

进入 MySQL：

```bash
mysql
```

创建数据库和用户：

```sql
CREATE DATABASE learning_platform CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'lms_user'@'localhost' IDENTIFIED BY '请换成一个强密码';
GRANT ALL PRIVILEGES ON learning_platform.* TO 'lms_user'@'localhost';
FLUSH PRIVILEGES;
EXIT;
```

记住这里的数据库密码，后面要写进 `server/.env`。

### 4. 下载项目代码

如果服务器能直接访问 GitHub，可以用：

```bash
git clone https://github.com/maobukeai/3d-personal-learning-platform.git /var/www/3d-lms
cd /var/www/3d-lms
```

如果 GitHub 连接不稳定，使用加速节点下载：

```bash
cd /var/www
wget -T 20 -O update.zip "https://gh-proxy.com/https://github.com/maobukeai/3d-personal-learning-platform/archive/refs/heads/main.zip"
unzip -q update.zip
rsync -a 3d-personal-learning-platform-main/ /var/www/3d-lms/
rm -rf update.zip 3d-personal-learning-platform-main
cd /var/www/3d-lms
```

### 5. 配置前端环境变量

编辑根目录 `.env`：

```bash
nano /var/www/3d-lms/.env
```

写入：

```env
VITE_API_URL=https://lilan8.cn
```

如果你还没配置 HTTPS，可以临时写：

```env
VITE_API_URL=http://你的服务器公网 IP:3001
```

### 6. 配置后端环境变量

编辑 `server/.env`：

```bash
nano /var/www/3d-lms/server/.env
```

推荐最小配置：

```env
PORT=3001
NODE_ENV=production

DATABASE_URL="mysql://lms_user:请换成你的数据库密码@localhost:3306/learning_platform"

JWT_SECRET="请换成一串很长的随机字符串"
DATABASE_ENCRYPTION_KEY="请换成 32 字节以上的随机字符串"

FRONTEND_URL=https://lilan8.cn
BACKEND_URL=https://lilan8.cn

ADMIN_INITIAL_PASSWORD="请换成首次管理员密码"
USER_INITIAL_PASSWORD="请换成测试用户密码"
```

可以用下面命令生成随机密钥：

```bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
```

`JWT_SECRET` 和 `DATABASE_ENCRYPTION_KEY` 都要长期保存，后续不要随便更换。

### 7. 首次构建和启动

给脚本执行权限：

```bash
cd /var/www/3d-lms
chmod +x deploy.sh fast_update.sh
```

执行首次部署：

```bash
NODE_BUILD_MEMORY_MB=1536 ./deploy.sh
```

说明：

- `deploy.sh` 会自动检查 / 创建 swap。
- 默认会跳过服务器上的 `vue-tsc`，只执行 `vite build`，避免 2G 小内存服务器爆内存。
- 如果服务器内存充足，可以用 `SKIP_TYPECHECK=0 ./deploy.sh` 强制完整类型检查。

## 三、第一次管理员账号怎么初始化

项目不会在服务启动时自动创建管理员。第一次部署完成后，需要手动初始化管理员一次。

推荐使用下面这段命令。它是幂等的：管理员不存在就创建，已存在就确保它是 `ADMIN` 且未封禁；同时会补一个个人空间。

```bash
cd /var/www/3d-lms/server

ADMIN_EMAIL="admin@example.com" \
ADMIN_INITIAL_PASSWORD="请换成你的首次管理员密码" \
node <<'NODE'
const { PrismaClient } = require('@prisma/client');
const bcrypt = require('bcryptjs');

const prisma = new PrismaClient();

async function main() {
  const email = process.env.ADMIN_EMAIL || 'admin@example.com';
  const password = process.env.ADMIN_INITIAL_PASSWORD;

  if (!password || password.length < 12) {
    throw new Error('请设置至少 12 位的 ADMIN_INITIAL_PASSWORD');
  }

  const hashedPassword = await bcrypt.hash(password, 10);

  const admin = await prisma.user.upsert({
    where: { email },
    update: {
      role: 'ADMIN',
      status: 'ACTIVE',
      emailVerified: true,
    },
    create: {
      email,
      password: hashedPassword,
      name: '系统管理员',
      role: 'ADMIN',
      status: 'ACTIVE',
      emailVerified: true,
    },
  });

  const personalTeam = await prisma.team.findFirst({
    where: {
      ownerId: admin.id,
      type: 'PERSONAL',
    },
  });

  if (!personalTeam) {
    await prisma.team.create({
      data: {
        name: `${admin.email} 的个人空间`,
        description: '个人专属创作与协作空间',
        type: 'PERSONAL',
        visibility: 'PRIVATE',
        ownerId: admin.id,
        members: {
          create: {
            userId: admin.id,
            role: 'OWNER',
          },
        },
      },
    });
  }

  console.log(`管理员已就绪：${admin.email}`);
}

main()
  .catch((error) => {
    console.error(error);
    process.exit(1);
  })
  .finally(async () => {
    await prisma.$disconnect();
  });
NODE
```

初始化完成后，打开网站登录：

```text
账号：admin@example.com
密码：你刚才设置的 ADMIN_INITIAL_PASSWORD
```

首次登录后建议立刻在前台个人设置里修改密码，或者在管理中心创建新的管理员账号。

### 关于项目里的 seed.ts

仓库里有 `server/src/seed.ts`，历史上用于创建默认管理员、默认用户和一些示例课程 / 资源。

但它不会在服务启动时自动执行，也不建议生产环境随手运行。生产服务器第一次部署时，推荐使用上面的轻量初始化命令，只创建必要的管理员账号和个人空间。

如果你明确想导入演示数据，再单独评估是否运行 seed。

### 如果忘记管理员密码

可以在服务器上重置：

```bash
cd /var/www/3d-lms/server

ADMIN_EMAIL="admin@example.com" \
NEW_ADMIN_PASSWORD="请换成新的强密码" \
node <<'NODE'
const { PrismaClient } = require('@prisma/client');
const bcrypt = require('bcryptjs');

const prisma = new PrismaClient();

async function main() {
  const email = process.env.ADMIN_EMAIL || 'admin@example.com';
  const password = process.env.NEW_ADMIN_PASSWORD;

  if (!password || password.length < 12) {
    throw new Error('请设置至少 12 位的新密码');
  }

  const hashedPassword = await bcrypt.hash(password, 10);
  await prisma.user.update({
    where: { email },
    data: {
      password: hashedPassword,
      role: 'ADMIN',
      status: 'ACTIVE',
      emailVerified: true,
    },
  });

  console.log(`管理员密码已重置：${email}`);
}

main()
  .catch((error) => {
    console.error(error);
    process.exit(1);
  })
  .finally(async () => {
    await prisma.$disconnect();
  });
NODE
```

## 四、以后怎么一键热部署

以后日常更新只需要两步。

### 1. 本地提交并推送代码

```bash
git add .
git commit -m "feat: 更新功能"
git push origin main
```

### 2. 服务器一键热部署

```bash
cd /var/www/3d-lms
./fast_update.sh
```

`fast_update.sh` 的逻辑：

1. 先把自己复制到 `/tmp` 再运行，避免脚本更新时覆盖正在执行的脚本。
2. 依次尝试多个备用 GitHub 加速节点下载最新 `main.zip`。
3. 解压到临时目录。
4. 用 `rsync` 安全覆盖项目代码。
5. 保留 `.env`、`server/.env`、`node_modules`、`server/node_modules`、`logs`、`server/uploads`。
6. 调用 `deploy.sh` 完成构建、数据库同步和 PM2 平滑重载。

正常提示类似：

```bash
================================================
📥 [1/3] 开始尝试通过多个备用加速节点下载最新代码...
-> 尝试连接节点...
✅ 下载成功！
📦 [2/3] 正在解压并安全覆盖旧代码...
🚀 [3/3] 代码覆盖完成！开始拉起低内存热部署流程...
================================================
```

## 五、和旧版热部署有什么不一样

旧版做法：

```text
下载 zip
解压
cp -r 直接覆盖当前目录
删除 migrations
sed 修改 deploy.sh
把 migrate deploy 替换成 db push --accept-data-loss
执行 deploy.sh
```

新版做法：

```text
下载 zip
解压到临时目录
rsync 安全覆盖
保留环境变量、上传文件、日志和依赖目录
不再在线 sed 修改部署脚本
不再使用 --accept-data-loss
自动准备 swap
默认跳过服务器 vue-tsc
最后 PM2 reload
```

新版最重要的改动是安全性：

- 不会覆盖正在运行的 `fast_update.sh`。
- 不会误删 `.env`、上传文件和日志。
- 不会用 `--accept-data-loss` 自动执行可能丢数据的数据库操作。
- 小内存服务器更不容易因为构建被系统 OOM 杀掉。

## 六、数据库结构变更怎么发布

如果只是新增字段、新增表，通常可以直接：

```bash
cd /var/www/3d-lms
./fast_update.sh
```

`deploy.sh` 会执行：

```bash
npx prisma db push --skip-generate
```

注意：

- 当前热部署脚本不使用 `--accept-data-loss`。
- 如果 Prisma 提示可能丢数据，脚本会停下来，这是正确的。
- 删除字段、改字段类型、重命名字段这类高风险变更，需要先备份数据库，再手动确认迁移方案。

备份 MySQL：

```bash
mysqldump -u lms_user -p learning_platform > ~/learning_platform_$(date +%F_%H%M).sql
```

## 七、新增环境变量怎么处理

`.env` 和 `server/.env` 不会被 Git 管理，也不会被热部署覆盖。

如果新增了环境变量，需要手动编辑服务器文件：

```bash
nano /var/www/3d-lms/server/.env
```

保存后执行：

```bash
cd /var/www/3d-lms
./fast_update.sh
```

如果只是后端环境变量变化，也可以不重新构建前端，直接重载：

```bash
cd /var/www/3d-lms
pm2 reload ecosystem.config.cjs --update-env
pm2 save
```

## 八、Nginx 配置与静态资源托管优化

### 1. 配置文件目录与启用方式（解决默认欢迎页冲突）

在首次部署时，如果直接用公网 IP 访问看到了 Nginx 默认的 "Welcome to nginx!" 页面，是因为默认站点占用了 80 端口，且未启用本项目配置。

**对于 Ubuntu / Debian 系统**：

1. 删除默认站点配置文件启用软链接（以释放 80 端口）：
   ```bash
   sudo rm -f /etc/nginx/sites-enabled/default
   ```
2. 新建本项目的配置文件：
   ```bash
   sudo nano /etc/nginx/sites-available/3d-lms
   ```
   （贴入下方的 Nginx 配置文件内容）
3. 启用该配置文件：
   ```bash
   sudo ln -sf /etc/nginx/sites-available/3d-lms /etc/nginx/sites-enabled/3d-lms
   ```

**对于 CentOS / RHEL / 阿里龙蜥等系统**：

- 直接在 `/etc/nginx/conf.d/` 目录下新建本项目的配置文件即可，无需做上述第一步和第三步：
  ```bash
  sudo nano /etc/nginx/conf.d/3d-lms.conf
  ```

***

### 2. Nginx 配置文件内容（已优化 3D 资产静态托管与缓存）

```nginx
server {
    listen 80;
    # ⚠️ 请将此处替换为你的域名（例如 lilan8.cn www.lilan8.cn）
    # 如果暂时没有域名只想用公网 IP 访问，可以写你的服务器 IP，或者直接写 `_` 匹配所有未绑定的访问
    server_name lilan8.cn www.lilan8.cn;

    root /var/www/3d-lms/dist;
    index index.html;

    # 上传文件限制大小（3D 模型文件通常比较大，默认 1M 会报 413 错误）
    client_max_body_size 500M;

    # -------------------------------------------------------
    # 1. 前端静态网页（Vue Router 历史模式防 404）
    # -------------------------------------------------------
    location / {
        try_files $uri $uri/ /index.html;
    }

    # -------------------------------------------------------
    # 2. 静态托管用户上传的 3D 模型与资产（优化大文件加载）
    # ⚠️ 必须使用 ^~ 前缀，确保优先级，避免 .png/.glb 匹配下方的静态规则导致 404
    # -------------------------------------------------------
    location ^~ /uploads/ {
        alias /var/www/3d-lms/server/uploads/;
        expires 7d;
        add_header Cache-Control "public";
        # 安全性：禁止在 uploads 目录中执行任何脚本文件
        location ~ \.(php|py|sh|cgi)$ {
            deny all;
        }
    }

    # -------------------------------------------------------
    # 3. 静态资源缓存优化（JS/CSS/图片/3D模型文件）
    # -------------------------------------------------------
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2|woff|ttf|glb|gltf|bin|hdr)$ {
        root /var/www/3d-lms/dist;
        expires 30d;
        add_header Cache-Control "public, no-transform";
        try_files $uri =404;
    }

    # -------------------------------------------------------
    # 4. 代理后端 API 请求 (Node.js Express)
    # -------------------------------------------------------
    location /api {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # 适当增加上传大文件时的连接与传输超时时间
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
        proxy_connect_timeout 75s;
    }

    # -------------------------------------------------------
    # 5. 代理 WebSocket (Socket.io 实时协作)
    # -------------------------------------------------------
    location /socket.io/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        proxy_read_timeout 86400s;
    }
}
```

### 3. 测试与重新加载

```bash
sudo nginx -t             # 测试配置文件语法
sudo systemctl reload nginx # 重新加载 Nginx 配置使其生效
```

### 4. 配合环境变量配置提醒

如果启用了以上 Nginx 反向代理，**必须**确保后端的 `server/.env` 中配置了 `TRUST_PROXY=1`。
否则，Express 的速率限制器（Rate Limiting）会错误地将 Nginx 本地 IP（127.0.0.1）当作所有用户的真实 IP，导致不同用户共享同一个限流配额，引发大面积 429 请求过多错误。

```env
TRUST_PROXY=1
```

HTTPS 证书可以用你熟悉的面板、云厂商证书，或 `certbot` 配置。

## 十、常见问题

### 1. `vue-tsc` 报 JavaScript heap out of memory

小内存服务器上很常见。热部署默认跳过 `vue-tsc`，只跑 `vite build`。

如果你手动开启了完整检查，可以改回：

```bash
SKIP_TYPECHECK=1 NODE_BUILD_MEMORY_MB=1536 ./deploy.sh
```

如果服务器上还停留在旧版 `deploy.sh`，还没有内置 `SKIP_TYPECHECK`，可以用下面这组命令临时救火：

```bash
cd /var/www/3d-lms

# 临时把前端构建从 vue-tsc + vite build 改成只跑 vite build
# 小内存热部署先跳过服务器上的类型检查
sed -i '0,/npm run build/s//npx vite build/' deploy.sh

# swap 已经有了，这次给 Node 1.5G
NODE_BUILD_MEMORY_MB=1536 ./deploy.sh
```

这组命令的作用：

- `sed -i '0,/npm run build/s//npx vite build/' deploy.sh`：只替换 `deploy.sh` 里第一次出现的 `npm run build`，也就是前端构建那一步，让它绕过 `vue-tsc`，直接执行 Vite 打包。
- `NODE_BUILD_MEMORY_MB=1536 ./deploy.sh`：给 Node 构建进程 1.5G 堆内存，避免之前 `768MB` 太低导致 `Reached heap limit`。

注意：这是临时修旧脚本用的命令。等新版 `deploy.sh` 和 `fast_update.sh` 推到 GitHub 并部署到服务器后，不需要再手动执行这条 `sed`。

### 2. SSH 突然断开

先查是不是系统 OOM：

```bash
dmesg -T | grep -Ei "out of memory|oom|killed process"
```

然后确认 swap：

```bash
free -h
swapon --show
```

### 3. 登录不了管理员

优先检查三件事：

```bash
cd /var/www/3d-lms/server
npx prisma studio
```

确认：

- `User` 表里有 `admin@example.com`。
- `role` 是 `ADMIN`。
- `status` 是 `ACTIVE`。

如果密码忘了，按本文“三、如果忘记管理员密码”重置。

### 4. 前端能打开，但 API 失败

检查：

```bash
pm2 list
pm2 logs 3d-lms-api --lines 100
curl -I http://127.0.0.1:3001/api/auth/settings
```

如果本机 API 正常，多半是 Nginx 反代或 `VITE_API_URL` 配置问题。

### 5. 热部署下载失败

`fast_update.sh` 会自动尝试多个备用节点。如果全失败，等几分钟再试，或者临时换用 Git：

```bash
cd /var/www/3d-lms
git pull origin main
./deploy.sh
```

### 6. `rsync` 出现大量 `file has vanished`

如果看到类似：

```text
file has vanished: "/var/www/3d-lms/.update_extract/..."
rsync warning: some files vanished before they could be transferred (code 24)
```

这是旧版 `fast_update.sh` 的临时目录放在了项目目录内部，`rsync --delete` 同步时把自己的解压目录删掉了。

先不要继续跑旧的 `./fast_update.sh`，用下面命令手动恢复一次：

```bash
cd /var/www/3d-lms

rm -rf /tmp/3d-lms-update update.zip .update_extract
mkdir -p /tmp/3d-lms-update

wget -T 20 -O /tmp/3d-lms-update/update.zip \
  "https://gh-proxy.com/https://github.com/maobukeai/3d-personal-learning-platform/archive/refs/heads/main.zip"

unzip -q /tmp/3d-lms-update/update.zip -d /tmp/3d-lms-update

rsync -a --delete \
  --exclude '.git/' \
  --exclude '.env' \
  --exclude 'server/.env' \
  --exclude 'node_modules/' \
  --exclude 'server/node_modules/' \
  --exclude 'logs/' \
  --exclude 'server/uploads/' \
  /tmp/3d-lms-update/3d-personal-learning-platform-main/ \
  /var/www/3d-lms/

chmod +x /var/www/3d-lms/deploy.sh /var/www/3d-lms/fast_update.sh

if ! grep -q 'SKIP_TYPECHECK' /var/www/3d-lms/deploy.sh; then
  sed -i '0,/npm run build/s//npx vite build/' /var/www/3d-lms/deploy.sh
fi

SKIP_TYPECHECK=1 NODE_BUILD_MEMORY_MB=1536 /var/www/3d-lms/deploy.sh
```

恢复完成后，把新版 `fast_update.sh` 推送到 GitHub。新版脚本会把临时解压目录放在 `/tmp/3d-lms-update-*`，不会再删到自己。

## 十一、推荐日常流程

本地开发：

```bash
npm run build
cd server
npm run build
cd ..
git add .
git commit -m "feat: 更新功能"
git push origin main
```

服务器更新：

```bash
cd /var/www/3d-lms
./fast_update.sh
```

这就是推荐的节奏：本地保证代码质量，服务器一键拉取、构建、同步数据库并平滑重载。
